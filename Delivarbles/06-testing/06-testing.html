1. Testing Strategy

The GameStore platform employs a single-layer service-level testing strategy with 185 total tests across 17 test files, covering all 17 BLL services. Every test targets the business logic layer in isolation using EF Core InMemory database and Moq for cross-service dependencies. There are no controller, integration, or E2E tests.

4. Sample Test Code

All tests follow the Arrange-Act-Assert pattern using EF Core InMemory database for data isolation and FluentAssertions for readable assertions. Below is a representative example from GameServiceTests.cs:

using FluentAssertions;

namespace GameStore.Tests.Services;

public class GameServiceTests
{
    private static GameStoreDbContext CreateContext(string dbName)
    {
        var options = new DbContextOptionsBuilder<GameStoreDbContext>()
            .UseInMemoryDatabase(dbName)
            .Options;
        return new GameStoreDbContext(options);
    }

    [Fact]
    public async Task GetAllWithCategoriesAsync_Returns_All_Games()
    {
        using var ctx = CreateContext("Game_GetAll");
        ctx.Games.AddRange(
            new Game { Id = "g1", Title = "Game A", ReleaseDate = DateTime.UtcNow, Price = 10m },
            new Game { Id = "g2", Title = "Game B", ReleaseDate = DateTime.UtcNow, Price = 20m }
        );
        await ctx.SaveChangesAsync();
        var uow = new UnitOfWork(ctx);
        var service = new GameService(uow);

        var games = await service.GetAllWithCategoriesAsync();

        games.Should().HaveCount(2);
    }

    [Fact]
    public async Task GetPagedAsync_Returns_Paginated_Games()
    {
        using var ctx = CreateContext("Game_Paged");
        for (var i = 1; i <= 10; i++)
            ctx.Games.Add(new Game { Id = $"g{i}", Title = $"Game {i}",
                ReleaseDate = DateTime.UtcNow, Price = i });
        await ctx.SaveChangesAsync();
        var uow = new UnitOfWork(ctx);
        var service = new GameService(uow);

        var result = await service.GetPagedAsync(1, 3);

        result.Items.Should().HaveCount(3);
        result.TotalCount.Should().Be(10);
        result.TotalPages.Should().Be(4);
    }
}

5. Mocking Strategy

Tests use EF Core InMemory database as the primary isolation mechanism, creating a fresh in-memory context per test with a unique database name to prevent state leakage. Moq is used to mock cross-service dependencies such as IGameAccessService, ISaleService, and ILibraryService when a service under test delegates to another service. IUnitOfWork is always the real UnitOfWork wrapping the in-memory context, ensuring transaction coordination is tested end-to-end. Dependencies are injected via constructor, following the real application's DI pattern.

7. Bug Reports

The following bugs were identified during static code analysis of the actual codebase. Each entry includes the real file location, severity, and recommended fix.

IDSeverityDescriptionLocationRecommended Fix
BUG-01HIGHStatic JsonSerializerOptions instance reused across threads in JsonFileStore.WriteAsync. The options object is not documented as thread-safe for serialization, risking data corruption under concurrent write load.Services/JsonFileStore.cs:10Create a new JsonSerializerOptions instance per WriteAsync call, or use Lock around serialization.
BUG-02HIGHHard-coded password salt fallback "GameStoreSalt2026" used when PasswordSalt config is missing. Attackers who know this default can perform dictionary attacks against legacy SHA256 password hashes.BLL/Services/AuthService.cs:15Remove the fallback value; throw InvalidOperationException if PasswordSalt is not configured.
BUG-03MEDIUMConnectionTracker.AddOrUpdate has a race condition on the first connection for a user: both factory delegates can execute concurrently, causing one connection ID to overwrite the other.Hubs/ConnectionTracker.cs:10Use GetOrAdd + lock on the returned HashSet, or use a ConcurrentDictionary with per-user semaphore.
BUG-04MEDIUMCreditService read-modify-write pattern is not atomic. The JsonFileStore semaphore is released between GetLedgerAsync and WriteLedgerAsync, allowing concurrent writes to overwrite each other.Services/CreditService.cs:46-57Wrap the read-modify-write sequence in a per-user SemaphoreSlim.
BUG-05MEDIUMStripe webhook catch block returns BadRequest("Invalid signature.") with no logging. Attackers probing the endpoint leave no trace.Controllers/StripeWebhookController.cs:37-40Add ILogger.LogWarning before returning the error response.
BUG-06MEDIUMRaw exception message (ex.Message) stored in admin notes and returned to callers, potentially leaking SQL errors, file paths, or stack traces to non-admin users.Services/RefundService.cs:115Log the full exception; return a sanitized generic message to callers.
BUG-07LOWEnum.Parse<Role>(roleStr) called without validation. If session Role is null or corrupted, a runtime ArgumentException is thrown.Controllers/LibraryController.cs:46,77Use Enum.TryParse<Role> with a default fallback.
BUG-08LOWVersion upload POST endpoints in Admin area lack [ValidateAntiForgeryToken], making them vulnerable to CSRF attacks.Areas/Admin/Controllers/GamesController.cs:228-251Add [ValidateAntiForgeryToken] attribute to all POST actions.
