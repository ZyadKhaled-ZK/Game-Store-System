Project Planning — GameStore Platform

Document version 1.0 — May 2026

1. Executive Summary

GameStore is a digital game distribution platform built on ASP.NET Core 8.0 (.NET 8) using the Model-View-Controller (MVC) architectural pattern. The platform provides a complete marketplace for purchasing, managing, and reviewing digital games, with role-based access for customers, developers, and administrators. The solution is structured across four projects: GameStore.PL (presentation layer with Razor views and controllers), GameStore.BLL (business logic layer with 20 service interfaces), GameStore.DAL (data access layer with Entity Framework Core 9.0.8 and SQL Server), and GameStore.Tests (unit tests using xUnit, Moq, and FluentAssertions, comprising 185 tests across 17 test files).

The platform implements cookie-based authentication (.GameStore.Auth cookie) with Google OAuth integration, Stripe payment processing (Checkout Sessions and webhooks), real-time features via SignalR (/hub/notifications hub handling notifications and chat), and SendGrid for transactional email delivery. Caching is handled through ASP.NET Core Output Cache (10-minute sliding expiration) with Redis fallback to in-memory distributed cache. The application is hosted on Kestrel (localhost:5000) with SQL Server and Redis provisioned via Docker Compose.

The project spans a 12-week delivery timeline from May through July 2026, organized into six phases. With 33 controllers (17 main, 11 Admin area, 5 Developer area), 62+ Razor views across three areas, and 23 database entities managed through 20 EF Core migrations, the platform delivers a comprehensive feature set including game browsing, shopping cart, wishlist, library management, developer dashboards, admin panels, chat/messaging, and a support ticket system.

2. Business Objectives

The following strategic objectives define the measurable goals for the GameStore platform launch and operation.

#Objective
1Deliver a production-ready ASP.NET Core MVC platform with full game catalog management, shopping cart, and checkout flow integrated with Stripe payments.
2Implement role-based access control supporting three tiers (CUSTOMER, DEVELOPER, ADMIN) with authorization policies (AdminOnly, DeveloperOnly, CustomerOnly) and granular area-based views.
3Achieve 185 unit tests across 17 test files covering all BLL service interfaces, using xUnit with FluentAssertions and Moq, with a minimum of 80% code coverage on business logic.
4Maintain platform availability of 99.9% uptime with Brotli/Gzip response compression, output caching, Redis-backed distributed caching, and rate limiting on auth endpoints.
5Provide a developer self-service portal enabling game uploads, sales management, revenue tracking, and review monitoring through dedicated Developer area views.
6Establish real-time communication features including friend requests, direct messaging, presence tracking, and notification delivery via SignalR hub.

Detailed risk register identifying potential project risks along with probability, impact, and planned mitigation measures.

RiskProbabilityImpactMitigation Strategy
Stripe webhook delivery failure or signature validation errors Medium High Implement webhook idempotency with Stripe.EventId; configure retry logic in StripeWebhookController; log all webhook payloads for manual reconciliation
SendGrid email delivery delays or bounces affecting auth flow (email verification, password reset) Medium High Implement email sending retry with exponential backoff; monitor SendGrid activity API; provide manual resend option in UI with rate limit (3/15min)
EF Core migration conflicts from schema changes across 20 migrations Low High Use migration history snapshots; test all migrations against staging DB before production; maintain rollback scripts for each migration
Redis cache connection failure causing performance degradation Low Medium Implement fallback to in-memory distributed cache when Redis is unavailable; monitor Redis health via docker-compose healthcheck
Rate limiting blocking legitimate users under peak load Medium Medium Set conservative rate limits (Login 10/min, Register 3/10min, Resend 3/15min, Verify 10/min); monitor 429 responses; whitelist known IPs for beta testing
Cookie authentication session hijacking via XSS Low Critical Enforce HttpOnly, Secure, SameSite=Lax on all cookies (.GameStore.Auth, .GameStore.Session); implement anti-forgery token on all POST requests; set X-Content-Type-Options and X-Frame-Options headers
SignalR connection churn under high concurrent chat usage Medium Medium Implement ConnectionTracker for connection lifecycle management; limit message payload size; configure Azure SignalR Service for scale in production
Game file upload failures (100 MB limit exceeded or disk space) Low High Enforce 100 MB request body limit in Kestrel + FormOptions; validate file type server-side; implement chunked upload for larger files; monitor disk usage on gamefile volume
Google OAuth breaking changes or callback URL mismatches Low Medium Register correct callback path (/auth/google-callback) in Google Console; implement error logging in external login flow; provide local account fallback
Brotli/Gzip compression causing response corruption on legacy proxies Low Low Enable compression only for known MIME types; test compressed responses across browsers; provide Accept-Encoding negotiation
