2. Key Implementation Details

2.1 Authentication Flow

The platform uses Cookie-based authentication (not JWT) with ASP.NET Core's built-in CookieAuthenticationDefaults scheme. Users authenticate via email/password or Google OAuth 2.0. Sessions are stored server-side with an 8-hour sliding expiration. The authentication pipeline includes a custom email verification middleware that blocks unverified users from accessing non-auth routes.

builder.Services.AddAuthentication(CookieAuthenticationDefaults.AuthenticationScheme)
    .AddCookie(options => {
        options.LoginPath        = "/Auth/Login";
        options.LogoutPath       = "/Auth/Logout";
        options.AccessDeniedPath = "/Home/Index";
        options.Cookie.Name      = ".GameStore.Auth";
        options.Cookie.HttpOnly  = true;
        options.Cookie.SameSite  = SameSiteMode.Lax;
        options.Cookie.SecurePolicy = CookieSecurePolicy.Always;
        options.ExpireTimeSpan   = TimeSpan.FromHours(8);
        options.SlidingExpiration = true;
    })
    .AddGoogle(options => {
        options.ClientId     = builder.Configuration["Google:ClientId"] ?? "";
        options.ClientSecret = builder.Configuration["Google:ClientSecret"] ?? "";
        options.CallbackPath = "/auth/google-callback";
        options.SignInScheme = CookieAuthenticationDefaults.AuthenticationScheme;
        options.SaveTokens   = true;
        options.Scope.Add("profile");
        options.Scope.Add("email");
    });

Password hashing uses BCrypt.Net-Next 4.0.3. Email verification tokens are stored in a dedicated EmailVerificationTokens table with expiry. A TokenCleanupService background service periodically purges expired tokens.

2.2 Shopping Cart Flow

The shopping cart is stored in the database via the Carts and CartItems tables, linked to the authenticated user. The CartController provides Add, Remove, and GetCount endpoints. Free games bypass Stripe and are processed via CompleteFreeCheckoutAsync which directly adds the game to the user's library. Paid games proceed through Stripe Checkout Sessions.

2.3 Stripe Payment Integration

Payments are processed through Stripe's Checkout API (Stripe.net 52.1.0). The CartController.Index creates a Stripe CheckoutSession with line items representing cart contents plus a success/cancel URL pair. The StripeWebhookController listens for checkout.session.completed events to finalize orders. A per-order commission percentage (stored in CommissionPercent on OrderItems) tracks the platform's revenue share for developer payouts.

var options = new SessionCreateOptions {
    PaymentMethodTypes = new List<string> { "card" },
    LineItems = cartItems.Select(item => new SessionLineItemOptions {
        PriceData = new SessionLineItemPriceDataOptions {
            Currency = "usd",
            ProductData = new SessionLineItemPriceDataProductDataOptions {
                Name = item.Game.Title
            },
            UnitAmount = (long)(effectivePrice * 100)
        },
        Quantity = 1
    }).ToList(),
    Mode = "payment",
    SuccessUrl = $"{domain}/cart/success?session_id={{CHECKOUT_SESSION_ID}}",
    CancelUrl = $"{domain}/cart"
};
var session = await new SessionService().CreateAsync(options);

2.4 SignalR Real-Time Notifications

Real-time notifications are delivered via a SignalR hub at the /hub/notifications endpoint. The NotificationHub (in GameStore.PL/Hubs/) manages connections via a ConnectionTracker singleton that maps userId to active connection IDs. When a notification is created (e.g., new support ticket, sale approval), the NotificationService persists the record to the UserNotifications table and broadcasts it to the recipient's active connections if online. The NotificationsController provides REST endpoints for fetching unread counts and marking notifications as read.

5. Code Architecture Patterns

5.1 Repository Pattern

Data access is abstracted behind the generic IRepository<T> interface (in GameStore.DAL/Repo/IRepository.cs) implemented by EfRepository<T>. Each entity has standard CRUD operations via the generic repository, while the GameStoreDbContext exposes 21 DbSet<T> properties for direct LINQ queries when needed. Domain-specific services compose repository calls within their methods.

5.2 Unit of Work

The IUnitOfWork interface (in GameStore.DAL/Repo/IUnitOfWork.cs) wraps the GameStoreDbContext and provides a Repository<T>() method to access the generic repository for any entity. It exposes SaveChangesAsync() to commit all pending changes atomically. Services inject IUnitOfWork directly, ensuring that operations spanning multiple tables (e.g., creating an order, adding items to the library, and clearing the cart) commit as a single transaction or roll back entirely on failure.

5.3 Dependency Injection

The platform leverages ASP.NET Core's built-in DI container. Services are registered in Program.cs with Scoped lifetime (one instance per HTTP request). The registration follows a consistent pattern: each service interface maps to its implementation. Infrastructure services like IJsonFileStore use Singleton lifetime (stateless JSON file access). Background services like TokenCleanupService use AddHostedService. Constructor injection is used throughout all controllers and services.

// Program.cs - Service registration (actual DI registrations)

7. Version Control & Branching Strategy

The project is hosted on GitHub with a GitFlow-inspired branching model. The CI pipeline (defined in .github/workflows/ci.yml) triggers on pushes to main and develop, and on pull requests targeting main.

BranchPurposeProtectionDeployment Target
mainProduction-ready code. Only merged via PR from develop or hotfix/*.Requires CI pass + code reviewProduction environment
developIntegration branch for completed features. All feature/* branches merge here.Requires CI passStaging environment
feature/*Individual feature work (e.g., feature/stripe-webhook, feature/signalr-chat).NoneDeveloper local
hotfix/*Urgent production fixes branched from main, merged back to both main and develop.Requires CI passProduction environment

Commit Convention: <type>(<scope>): <description> — e.g., feat(auth): add Google OAuth callback, fix(cart): resolve race condition in checkout, test(order): add CompleteCheckout unit tests.

The continuous integration pipeline runs on GitHub Actions (ubuntu-latest) with SQL Server 2022 and Redis 7-alpine as service containers. The pipeline executes on every push to main/develop and on every PR to main.

Configuration

All sensitive configuration is managed via .NET User Secrets in development and environment variables in production. The appsettings.json file contains only non-sensitive defaults. Key configuration sections:

  • ConnectionStrings:GameStoreConnection — SQL Server connection string
  • ConnectionStrings:Redis — Redis connection string (empty = in-memory cache fallback)
  • Stripe:SecretKey / Stripe:PublishableKey / Stripe:WebhookSecret — Stripe API credentials
  • SendGrid:ApiKey / SendGrid:FromEmail / SendGrid:FromName — Email provider
  • Google:ClientId / Google:ClientSecret — OAuth 2.0 credentials
  • PasswordSalt — Salt for legacy SHA256 password hashing
